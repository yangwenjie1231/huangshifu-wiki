本次已通读当前仓库的核心代码与配置，覆盖：

## 2. 当前已实现功能（详细）

### 2.1 架构与运行方式

- 前端：React 19 + TypeScript + Vite 6 + Tailwind 4
- 后端：Express（`server.ts`），开发模式下与 Vite middleware 同进程
- 数据库：MySQL + Prisma
- 鉴权：JWT（HttpOnly Cookie，支持 `Authorization Bearer` 兜底）
- 文件上传：本地 `uploads/` 目录 + 静态托管 `/uploads/*`

### 2.2 用户与认证

- 本地邮箱注册/登录/登出
  - `POST /api/auth/register`
  - `POST /api/auth/login`
  - `POST /api/auth/logout`
  - `GET /api/auth/me`
- 微信登录（含 mock 联调）
  - `POST /api/auth/wechat/login`
  - 支持 `WECHAT_LOGIN_MOCK=true` 和 `mock:openId[:unionId]`
- 用户资料更新
  - `PATCH /api/users/me`
- 角色体系
  - `user` / `admin` / `super_admin`
- 状态体系
  - `active` / `banned`，封禁用户会被拦截写操作

### 2.3 权限与治理

- 管理员权限中间件完整：`requireAdmin`、`requireSuperAdmin`
- 封禁/解封
  - `POST /api/admin/users/:uid/ban`
  - `POST /api/admin/users/:uid/unban`
  - 记录 `UserBanLog`
- 超管改角色
  - `PATCH /api/users/:uid/role`

### 2.4 Wiki（百科）

- 页面列表、分类筛选、详情、时间轴、推荐
  - `GET /api/wiki`
  - `GET /api/wiki/:slug`
  - `GET /api/wiki/timeline`
  - `GET /api/wiki/recommended`
- 可见性控制：游客仅 `published`，作者可见自己草稿，管理员全可见
- 新建/编辑/草稿/提交审核
  - `POST /api/wiki`
  - `PUT /api/wiki/:slug`
  - `POST /api/wiki/:slug/submit`
- 历史版本与回滚
  - `GET /api/wiki/:slug/history`
  - `POST /api/wiki/:slug/revisions`
  - `POST /api/wiki/:slug/rollback/:revisionId`
- 收藏系统（支持 Wiki）
- 统计字段：`viewCount`、`favoritesCount`
- 前端增强：Markdown 编辑器、内部链接 `[[slug]]` 解析、AI 摘要/AI 开头生成

### 2.5 Forum（社区）

- 版块系统
  - `GET /api/sections`
  - `POST /api/sections`（管理员）
  - `DELETE /api/sections/:id`（管理员）
- 帖子列表与排序
  - `GET /api/posts?sort=latest|hot|recommended`
- 帖子详情
  - `GET /api/posts/:id`
  - 自动累加 `viewCount`，并刷新 `hotScore`
- 发帖/提审
  - `POST /api/posts`
  - `POST /api/posts/:id/submit`
- 评论（含楼中楼 parentId）
  - `POST /api/posts/:id/comments`
- 点赞
  - `POST /api/posts/:id/like`
  - `DELETE /api/posts/:id/like`
- 收藏系统（支持帖子）
- 统计字段：`likesCount`、`commentsCount`、`hotScore`

### 2.6 通知与行为沉淀

- 通知查询与已读
  - `GET /api/notifications`
  - `POST /api/notifications/:id/read`
  - `POST /api/notifications/read-all`
- 触发来源（当前已打通）
  - 被评论/被回复
  - 帖子被点赞
- 浏览历史
  - `GET /api/users/me/history`
  - 30 分钟去重逻辑

### 2.7 搜索

- 综合搜索
  - `GET /api/search`
  - 支持 `q/type/category/startDate/endDate`
- 热搜
  - `GET /api/search/hot-keywords`
- 联想建议
  - `GET /api/search/suggest`
- 前端高级筛选：标签、时间范围、内容类型
- AI 图片搜索：上传图片后调用 Gemini 生成关键词再搜索

### 2.8 图集与图片映射

- 图集列表与创建
  - `GET /api/galleries`
  - `POST /api/galleries`
  - `DELETE /api/galleries/:id`（管理员）
- 文件上传
  - `POST /api/uploads`
  - `POST /api/galleries/upload`（后端已提供）
- 图片映射管理（CDN URL 映射）
  - `GET /api/image-maps`
  - `GET /api/image-maps/:id`
  - `POST /api/image-maps`
- 前端 `SmartImage` 支持多 URL 失败回退

### 2.9 音乐

- 音乐列表
  - `GET /api/music`
- 新增歌曲（管理员）
  - `POST /api/music`
  - `POST /api/music/from-netease`
- 删除歌曲（管理员）
  - `DELETE /api/music/:docId`
- 按 NetEase ID 获取单曲元数据
  - `GET /api/music/song/:id`
- 前端全局播放器（跨页面保持）、音乐收藏

### 2.10 审核后台（Admin）

- 审核队列
  - `GET /api/admin/review-queue`
  - `POST /api/admin/review/:type/:id/approve`
  - `POST /api/admin/review/:type/:id/reject`
- 内容管理：wiki / posts / galleries / users / sections / announcements / music
- 用户管理：升降权限、封禁解封
- 公告管理：新增、启停、删除

### 2.11 小程序接口（P2 已有后端能力）

- `GET /api/mp/wiki`
- `POST /api/mp/posts`
- `POST /api/mp/comments`

### 2.12 运维与部署

- 健康检查：`GET /api/health`
- 一键部署脚本：`scripts/deploy.sh`
- PM2 + Nginx + HTTPS 部署文档完整

## 3. 目前最需要测试的内容

> 当前仓库**没有测试框架与测试用例**（无 `test` script，无 `*.test.*` / `*.spec.*`）。
> 现阶段仅有 `npm run lint`（TypeScript 类型检查）。

下面按优先级给出“必须补测”清单：

### P0（上线稳定性必测）

1. 认证与会话
   - 注册/登录/登出/刷新页面保持登录
   - Cookie 在 HTTP/HTTPS 下的 `secure` 行为
   - 微信登录真实模式与 mock 模式

2. 权限边界
   - 游客不可写
   - 普通用户不可进 admin、不可改角色
   - 封禁用户不可发帖/评论/编辑
   - super_admin 与 admin 权限差异准确

3. 审核流闭环（wiki + post）
   - 草稿 -> 提审 -> 通过/驳回 -> 前台可见性变化
   - 驳回后再编辑再提审
   - 审核日志 `ModerationLog` 正确记录

4. 互动一致性
   - 点赞/取消点赞幂等（重复点击不脏数据）
   - 收藏/取消收藏幂等
   - 评论+回复后计数准确
   - viewCount/hotScore/favoritesCount 计数一致

5. 上传链路
   - 单图上传、图集上传、超大文件、非法类型、失败重试

### P1（体验与数据质量）

1. 搜索质量
   - 综合搜索、筛选、联想、热词计数
   - 空结果和慢查询场景

2. 通知
   - 点赞/回复触发
   - 单条已读/全部已读
   - 多端/多标签页一致性

3. 用户中心
   - 我的帖子/评论/历史/收藏分页与可见性

### P2（安全与回归）

1. 安全
   - 输入注入（XSS、超长输入、异常 JSON）
   - 上传安全（扩展名伪造、路径问题）

2. 回归
   - Home / Wiki / Forum / Search / Gallery / Music / Admin 全链路冒烟
   - 移动端和桌面端主要流程

## 4. 审阅中发现的缺口与风险点（建议优先修）

1. `review_result` 通知未实际触发
   - 后端定义了通知类型，前端也处理了该类型文案。
   - 但审核通过/驳回接口中未调用 `createNotification(...)`。
   - 结果：用户收不到“审核结果通知”。

2. 部分 UI 操作是占位行为
   - 音乐播放器上一首/下一首按钮无逻辑。
   - 帖子“分享”按钮无实际动作。

3. 搜索联想跳转使用 `window.location.href`
   - 会触发整页刷新，非 SPA 路由跳转。
   - 建议改 `useNavigate()`。

4. 首页部分数据为静态展示
   - Hero 图与“收录曲目/社区成员”数字是固定值，非真实统计。

5. 历史遗留配置文件与现架构并存
   - `firestore.rules` / `firebase-blueprint.json` / `firebase-applet-config.json` 仍存在。
   - 当前运行主链路已迁移到本地后端 API + Prisma，建议明确保留策略，避免误导。

## 5. 可新增功能建议（按投入产出排序）

### 5.1 低成本高收益（建议先做）

1. 打通审核结果通知
   - 审核通过/驳回时给作者发 `review_result`。

2. 完成“分享”与“上一首/下一首”
   - 分享先做“复制链接 + 提示”。
   - 歌曲切换先做当前列表内顺序切换。

3. 图集详情页
   - 当前更多是卡片流，建议增加 `/gallery/:id` 详情与大图浏览。

4. 用户可编辑自己的帖子
   - 目前主要是新建与审核，补齐“编辑已发布/草稿”会显著提升可用性。

### 5.2 中期增强（产品增长）

1. 小程序端最小前端
   - 既然后端已有 `/api/mp/*`，可补一个最小可用客户端闭环。

2. 作品库结构化（Album/Track）
   - 从“歌曲列表”升级为“作品数据库 + 关系模型”。

3. 搜索排序升级
   - 在关键词匹配基础上，叠加热度与时间衰减权重。

4. 通知中心独立页面
   - 现在仅 Navbar 下拉，建议扩展为完整通知页（筛选、批量、跳转追踪）。

### 5.3 工程能力建设（强烈建议）

1. 引入自动化测试体系
   - 后端：Vitest + Supertest（API 集成测试）
   - 前端：Playwright（关键路径 E2E）

2. 错误监控与审计
   - 接入 Sentry（前后端）
   - 管理操作审计日志可视化

3. 性能与容量
   - 列表分页统一
   - 热点接口缓存（如首页 feed、热搜）
